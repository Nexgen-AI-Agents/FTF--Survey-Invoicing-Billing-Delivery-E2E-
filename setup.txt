ARCHITECTURE OVERVIEW
─────────────────────
  Compute      : GitHub Actions (ubuntu-latest, Python 3.11)
  Database     : AWS RDS MySQL — nexgen_ftf_db
  State Store  : Git-committed Excel + JSON files (data/ directory)
  AI Engine    : Anthropic Claude API (primary) + OpenAI (fallback)
  Auth         : Azure AD / Microsoft Graph (OneDrive, Teams, Email)
  Scheduling   : GitHub Actions cron + Power Automate workflow_dispatch
  Storage      : GitHub repo + OneDrive (no S3 currently)

No EC2, Lambda, or ECS is used. GitHub Actions IS the compute layer.
AWS is used only for RDS MySQL.


================================================================================
STEP 1 — AWS RDS MYSQL SETUP
================================================================================

1.1  Create the RDS Instance
─────────────────────────────
  Engine           : MySQL 8.0+
  Instance class   : db.t3.small (minimum for staging), db.t3.medium for prod
  Storage          : 20 GB gp2 (auto-scaling enabled)
  Multi-AZ         : No (staging), Yes (prod)
  Publicly accessible : YES — GitHub Actions runners use public internet
  DB name          : nexgen_ftf_db
  Port             : 3306
  Username         : (your choice, e.g. nexgen_admin)
  Password         : (strong password, save for GitHub Secrets)

1.2  Security Group Rules
──────────────────────────
  Inbound rule:
    Type     : MySQL/Aurora
    Port     : 3306
    Source   : 0.0.0.0/0   ← OK for staging; tighten for prod
               OR use GitHub Actions IP ranges from:
               https://api.github.com/meta  (field: "actions")

  Outbound rule:
    Type     : All traffic
    Source   : 0.0.0.0/0

1.3  Database & User Setup
───────────────────────────
  Connect to RDS:
    mysql -h <your-rds-host>.rds.amazonaws.com -u nexgen_admin -p

  Run these SQL commands:

    CREATE DATABASE IF NOT EXISTS nexgen_ftf_db CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;

    CREATE USER 'ftf_pipeline'@'%' IDENTIFIED BY '<strong-password>';

    GRANT SELECT, INSERT, UPDATE, DELETE ON nexgen_ftf_db.* TO 'ftf_pipeline'@'%';

    FLUSH PRIVILEGES;

  Verify:
    SHOW GRANTS FOR 'ftf_pipeline'@'%';

  NOTE: The pipeline READS from the FTF API for order data. RDS MySQL is used for
  pipeline state tracking (approved orders, learned rules cache). The FTF app's
  own database is separate — managed by NexGen IT.

1.4  Verify Connectivity
─────────────────────────
  From any machine:
    curl -v telnet://<your-rds-host>.rds.amazonaws.com:3306

  Should return: "Connected to <host>"
  If it hangs: security group inbound rule is missing.

STEP 3 — AZURE APP REGISTRATION (OneDrive + Teams)
================================================================================

  Required for: reading/writing the Excel approval sheet on OneDrive,
                sending Teams notifications, reading Teams replies.

  3.1  In Azure Portal → Entra ID → App Registrations → New registration
        Name: FTF-Pipeline
        Redirect URI: none (service app)

  3.2  API Permissions (Application type, not Delegated):
        Microsoft Graph:
          Files.ReadWrite.All
          Mail.Send
          Mail.ReadWrite
          User.Read.All
          ChannelMessage.Send          (for Teams posting)
          ChannelMessage.Read.All      (for reading Teams replies)

        Click "Grant admin consent" — required for Application permissions

  3.3  Certificates & Secrets → New client secret
        Copy the VALUE immediately (shown only once) → AZURE_CLIENT_SECRET

  3.4  Overview → copy:
        Application (client) ID → AZURE_APP_ID
        Directory (tenant) ID   → AZURE_TENANT_ID

================================================================================
STEP 5 — POWER AUTOMATE TRIGGERS
================================================================================

  The pipeline does NOT run on a time schedule by itself.
  Power Automate calls GitHub's workflow_dispatch API to trigger runs.

  5.1  Invoice Pipeline (runs every 30 minutes):
        Power Automate → Scheduled flow → Recurrence: 30 min
        HTTP action:
          Method : POST
          URL    : https://api.github.com/repos/Nexgen-AI-Agents/FTF--Survey-Invoicing-Billing-Delivery-E2E-/actions/workflows/invoice_pipeline.yml/dispatches
          Headers: Authorization: Bearer <GitHub PAT with workflow scope>
                   Accept: application/vnd.github+json
          Body   : {"ref": "main"}

  5.2  Excel Approval Watcher (runs every 5 minutes):
        Same setup but target workflow: excel_approval_watcher.yml

  5.3  Test the trigger manually:
        GitHub → Repo → Actions → Invoice Pipeline → Run workflow → Run workflow (manual)

STEP 6 — LOCAL DEVELOPMENT SETUP
================================================================================

  Prerequisites:
    Python 3.11+ (not 3.12+ — TLS regression with FEMA API)
    Git
    Access to GitHub repo

  6.1  Clone the repo:
        git clone https://github.com/Nexgen-AI-Agents/FTF--Survey-Invoicing-Billing-Delivery-E2E-.git
        cd FTF--Survey-Invoicing-Billing-Delivery-E2E-

  6.2  Create virtual environment:
        python -m venv .venv
        .venv\Scripts\activate          (Windows)
        source .venv/bin/activate       (Linux/Mac)

  6.3  Install dependencies:
        pip install -r requirements.txt

  6.4  Create .env file:
        copy .env.example .env
        (Fill in all values — see Step 2 for what each secret is)

  6.5  Verify database connection:
        cd code/sprint_11_invoice_pipeline
        python -c "from code.shared.core.ftf_mysql import get_connection; c = get_connection(); print('DB OK'); c.close()"

  6.6  Run the pipeline manually:
        cd code/sprint_11_invoice_pipeline
        python -m agents.agent_a0_orchestrator

  6.7  Run tests:
        pytest code/ -v --tb=short

================================================================================
KNOWN GOTCHAS
================================================================================

  1. GitHub Actions IP ranges change periodically — if RDS connection fails,
     check https://api.github.com/meta and update the security group inbound rule.

  2. OneDrive Graph API token expires. The Azure App uses client credentials
     (no user login needed), so no manual refresh is required as long as
     AZURE_CLIENT_SECRET is not expired. Rotate it before the expiry date shown
     in Azure Portal.

  3. Python version MUST be 3.11. Python 3.12+ has a TLS regression that breaks
     the FEMA ArcGIS REST API calls. Pinned in all workflow files.

  4. The data/ directory files (pipeline_state.json, invoice_pipeline_state.xlsx)
     are committed to git after every run by the GitHub Actions bot. On concurrent
     runs the last-writer-wins. This is intentional — not a bug.

  5. EMAIL_OVERRIDE_ALL must be set during all staging/testing. Every invoice
     email (including customer-facing ones) gets redirected to that address.
     Removing it in staging by accident will send real emails to real customers.

  6. PAT (DASHBOARD_GITHUB_TOKEN) used for cross-repo dashboard pushes must have
     repo scope. Fine-grained PATs work only if explicitly granted write access to
     the target dashboard repo.

  7. APPROVED_SENDERS controls which email senders can trigger approval actions
     via email reply. Format: name:email,name:email — no spaces around commas.
